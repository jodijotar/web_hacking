#phase

NEW PHASE. the single biggest gap in pure-recon workflows for BAC/IDOR. without this phase there is no BAC testing — only unauthenticated misconfig hunting

account provisioning
	create 3 accounts per role tier:
		userA, userB  -> horizontal IDOR (same role)
		admin (if invite system permits)
	create 2 tenants/orgs where the program allows
		userA in tenant1, userC in tenant2 -> cross-tenant IDOR

cookie isolation
	PwnFox + Firefox Multi-Account Containers
		one container per account. visual color tagging. avoids cookie collision when testing replays manually
	Caido alternative: workspace per account, or use the Sessions plugin for per-request identity switching

session handlers
	Caido sessions / Burp session handlers
		auto-refresh tokens on 401/403. without this, hour-long authenticated scans break silently mid-run
	store creds in a vault file (NOT in this repo):
		`~/.bbp_creds/<target>.env` -> sourced by helper scripts

document identifiers per account
	for each role: record userID, tenantID, slugs, role markers, any "you can do X" indicators in the UI
	these are the seed values for the BAC/IDOR matrix in phase 7

invite/share/billing exploration (Douglas Day discipline)
	spend the first 30 minutes inside invite, sharing, billing, and team-management features
		invite an already-existing user (PII leak?)
		invite -> auto-acceptance without consent?
		modify the invite payload before sending (role escalation?)
		expired invite reuse?
		invite to a tenant you don't own (tenant takeover?)
	these features are where SaaS BAC bugs cluster
