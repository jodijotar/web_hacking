#phase

nuclei (tight tags — avoid dedup competition)
	`nuclei -l priority_hosts.txt -tags exposure,token,takeover -severity high,critical -rl 30 -o nuclei_findings.txt`
	the wider categories (info, tech, ssl, misc) compete with every automated scanner on the platform — duplicate-finder fodder
	the chosen tags target findings with actual chain potential:
		exposure -> exposed configs, internal panels, env files
		token -> validated leaked secrets via token-spray
		takeover -> dangling DNS records

custom nuclei templates (the real edge here)
	build target-specific templates for:
		internal hostnames discovered in JXScout bundles
		framework-specific endpoints from leaked openapi
		CVE-relevant signatures matching the live_hosts.csv tech-detect column
	store under `~/nuclei-templates-custom/<target>/`
	this is where you avoid the "everyone runs default templates" dedup problem

subdomain takeover sweep
	`subzy run --targets subdomains_resolved.txt --hide_fails --concurrency 50 -o subzy_takeovers.txt`
	`nuclei -l subdomains_resolved.txt -t http/takeovers/ -o nuclei_takeovers.txt`
	cross-check `can-i-take-over-xyz` for services not yet templated; if found unmatched -> write a custom nuclei template for it (Patrik Hudak's "first PoC wins")

cloud asset enumeration (one tool — don't stack)
	`cloud_enum -k <target_brand> -k <target_alt_name> --disable-azure-vm --disable-gcp-functions`
	covers AWS S3 + Azure blobs + GCS. permutation quality > tool count
	for any bucket found: test public read, public write, public list, dangling reference. dangling CloudFront distributions are high-impact and rarely reported

run as background scheduled job during phase 6/7 manual work. triage hits opportunistically — don't context-switch off the authenticated workflow
